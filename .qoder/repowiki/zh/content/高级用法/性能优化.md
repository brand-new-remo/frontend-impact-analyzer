## 简介
本指南聚焦于大规模前端项目的性能优化策略与实践，围绕内存使用、并发处理、缓存机制、AST 解析瓶颈与增量分析、配置参数调优（超时、并发、资源限制）、性能分析工具使用、不同项目规模的优化策略、监控与基准测试方法，以及在"分析精度"与"性能表现"之间的平衡进行系统阐述。文档以仓库中的实际实现为依据，结合可视化图示与分层讲解，帮助读者快速落地优化。

**更新** 本版本重点介绍阶段执行系统（Phased Execution）带来的性能改进：从单阶段全量分析转变为三阶段分步执行，支持自动和手动阶段执行模式，显著提升了大PR分析的性能表现。

## 项目结构

## 详细组件分析

### 组件一：阶段执行系统与性能改进
**更新** 阶段执行系统是本次性能优化的核心，实现了从单阶段全量分析到三阶段分步执行的根本性转变。

- **阶段执行模式**
  - **单阶段全量执行**：传统模式，一次性完成所有分析步骤
  - **自动阶段执行**：当diff行数超过analysis.phasedExecutionThreshold阈值时自动切换到分步执行
  - **手动阶段执行**：用户明确指定--phase参数，支持parse、scan、analyze三个阶段的独立执行
- **检查点管理**
  - parse阶段：解析diff、分类变更文件、写入phase-01-parse.json检查点
  - scan阶段：扫描源码、构建导入图、写入phase-02-scan.json检查点
  - analyze阶段：执行影响分析、聚类构建、输出最终结果
- **性能收益**
  - 大型PR分析时间显著缩短，内存占用大幅降低
  - 支持长时间运行的分析任务，避免单次执行超时
  - 提供更好的错误恢复能力，部分失败时可以重新执行特定阶段

```mermaid
flowchart TD
Start(["开始分析"]) --> CheckThreshold["检查 diff 行数"]
CheckThreshold --> |超过阈值| AutoPhase["自动阶段执行"]
CheckThreshold --> |未超过阈值| SinglePhase["单阶段全量执行"]
AutoPhase --> ParsePhase["阶段1：parse 阶段<br/>解析diff、分类文件、写入检查点"]

```

### 组件二：AST 解析与性能瓶颈
- 关键实现
  - 使用 Tree-sitter 的 TS/TSX 解析器，逐节点遍历并抽取所需信息。
  - 在单文件内进行正则与字段匹配，避免复杂回溯。
- 性能瓶颈
  - 文件数量与体积线性增加导致 CPU 与内存占用上升。
  - 大型项目中导入解析与别名展开可能产生重复 IO 与字符串操作。
- 优化建议
  - **阶段执行系统**：parse阶段仅解析diff，scan阶段才进行全量AST解析
  - 文件过滤：仅扫描 src 目录与受支持扩展名，忽略 node_modules 等目录。
  - 增量解析：基于变更文件集合，仅对受影响的导入链路重新解析。
  - 缓存 AST 事实：将已解析的 AST 事实按文件缓存，避免重复解析。
  - 并发解析：在多核环境下对独立文件进行并发解析（注意 GIL 限制，可考虑进程池）。
  - 语义标签派生：将高频模式抽取为常量，减少重复正则编译。

### 组件三：项目扫描与导入解析
- 关键实现
  - 遍历 src 或项目根目录，过滤扩展名与忽略目录。
  - 对每个文件调用 AST 分析器，解析导入/导出/懒加载与路由记录。
  - 解析别名目标，尝试多种候选后缀与 index 变体，建立导入/反向导入图。
  - 收集诊断（未解析导入、未绑定路由等）。
- 性能瓶颈
  - 递归解析别名与候选文件存在多次 IO。
  - 大量小文件导致频繁系统调用与字符串拼接。
- 优化建议
  - **阶段执行系统**：parse阶段构建完整导入图，scan阶段仅分析需要的文件
  - 引入"导入解析缓存"，命中则直接返回解析结果。
  - 将候选扩展名与 index 变体预构建为映射表，减少循环查找。
  - 对路由记录的展开采用"延迟求值"，仅在需要时解析具体文件。
  - **检查点复用**：parse阶段生成的导入图在scan阶段直接复用，避免重复解析。

### 组件四：影响分析与回溯追踪
- 关键实现
  - 基于反向导入图进行 BFS，从变更文件向上回溯至页面。
  - 符号传播：根据导入/再导出绑定与标识符计数，决定是否保留活跃符号。
  - 计算影响类型、置信度与原因，合并语义标签。
- 性能瓶颈
  - 大型依赖图中 BFS 可能产生大量中间 trace，内存与去重成本高。
  - 符号匹配与去重逻辑在长链路中重复计算。
- 优化建议
  - 使用"访问键去重"（当前实现已采用），确保 (文件, 活跃符号, 是否严格) 唯一。
  - 控制最大回溯层级或 hop 数，避免深层链路无限扩散。
  - 对页面集合使用集合查询，提升命中速度。

### 不同项目规模的优化策略

  - 强制阶段执行：当diff行数超过analysis.phasedExecutionThreshold阈值时自动切换到分步执行。
