## 简介
本指南面向高级用户，系统讲解前端影响分析器的高级用法与扩展策略，覆盖以下主题：
- 自定义分析规则的配置与扩展（语义标签、影响置信度、用例模板）
- 代理工具集成最佳实践（含 OpenAI 代理）
- 项目约定与路由约定对分析结果的影响与应对
- 性能优化技巧与大规模项目的处理策略
- **阶段执行工作流程与检查点管理**（新增）
- **大型diff文件处理策略**（新增）
- 环境诊断功能增强：虚拟环境冲突检测与诊断建议
- 故障排除与常见问题定位
- 扩展工具以支持新前端框架或特殊需求
- 监控与日志配置建议
- CI/CD 集成实施方案

## 核心组件
- 命令行入口与引擎：负责解析参数、组织分析流程、记录过程日志与状态、输出结果与诊断。
- 差异解析器：从 Git diff 中提取变更文件、格式化变更判断、语义标签与 API 变更。
- 项目扫描器：遍历源码、解析 AST、建立导入/导出关系、识别页面与路由、解析别名与条目。
- 影响分析引擎：基于反向导入图与路由映射，从变更文件追踪到页面，计算影响类型与置信度。
- 用例构建器：根据语义标签与 API 变更推导业务用例，生成结构化测试用例。
- 模型与状态存储：统一的数据结构、状态持久化与 Schema 校验。
- 通用工具：路径规范化、别名解析、去重、标题生成等。
- **工作流与环境诊断**：负责预检、环境诊断、虚拟环境隔离检查、依赖验证与推荐命令。
- **阶段执行功能**：支持独立的分析阶段运行，包括解析、扫描和分析三个阶段。
- **检查点管理**：通过phase-01-parse.json和phase-02-scan.json文件持久化中间结果。

## 详细组件分析

### 组件一：差异解析器（GitDiffParser）
- 功能要点
  - 提取变更文件、新增/删除/修改类型
  - 格式化变更快速判定（仅空白/引号变化）
  - 语义标签抽取（按钮、弹窗、表单、表格、路由、权限、API、状态、导航、校验、列表/详情/删除、上传、禁用态等）
  - API 变更分析（请求/响应字段重命名、枚举增删、分页/详情/列表结构变化）
- 关键实现位置
  - 符号与语义模式匹配、API 变更抽取与去重
  - 语义标签到 API 类别的映射
- 扩展建议
  - 在语义模式与 API 变更映射处增加新标签与类别
  - 结合项目约定与路由约定，细化语义识别

```mermaid
flowchart TD
Start(["开始解析 diff"]) --> Split["按文件切分块"]
Split --> Hunk["进入 hunk 区域"]
Hunk --> Add["+ 行收集与符号/语义提取"]
Hunk --> Remove["- 行收集与符号/语义提取"]
Add --> DoneAdd["汇总已添加行"]
Remove --> DoneRemove["汇总已删除行"]
DoneAdd --> CheckFmt{"是否格式化变更？"}
DoneRemove --> CheckFmt
CheckFmt --> |是| Clear["清空符号/语义/API 变更"]
CheckFmt --> |否| AnalyzeAPI["分析 API 字段/枚举/分页等变化"]
AnalyzeAPI --> MergeTags["合并语义标签"]
Clear --> End(["结束"])
MergeTags --> End
```

### 组件二：项目扫描器（ProjectScanner）
- 功能要点
  - 遍历源码文件，解析 AST，抽取导入/导出、组件名、Hook 名、JSX 标签/属性、路由对象、懒加载、API 调用、语义标签
  - 解析别名（tsconfig paths）与 barrel 导出证据
  - 识别页面（基于目录与组件/JSX）
  - 路由绑定：组件名、懒加载、父子路径拼接、未绑定路由诊断
- 关键实现位置
  - 路由记录提取与展开、页面猜测、导入解析候选
  - 别名解析与候选文件探测
- 扩展建议
  - 在别名解析处增加额外约定（如多级别名前缀）
  - 在页面识别处加入更多启发式规则

```mermaid
flowchart TD
S0["开始扫描"] --> Collect["收集源码文件"]
Collect --> ParseAST["逐文件解析 AST 并抽取事实"]
ParseAST --> ResolveImport["解析导入/导出与别名"]
ResolveImport --> Barrel["识别 barrel 文件与证据"]
Barrel --> PageCheck["判断页面文件"]
PageCheck --> RouteExtract["提取路由对象与懒加载"]
RouteExtract --> BindRoute["绑定路由到页面并记录诊断"]
BindRoute --> BuildGraph["构建导入/反向导入图"]
BuildGraph --> Output["返回图谱与诊断"]
```

### 阶段执行工作流程详解（新增）

#### 独立阶段运行
- **--phase 参数支持**
  - `--phase parse`：仅执行差异解析和文件分类
  - `--phase scan`：仅执行项目扫描和图谱构建
  - `--phase analyze`：仅执行影响分析和结果生成
- **--run-dir 参数**
  - 用于指定阶段执行的运行目录
  - 存储检查点文件和中间结果
- **检查点文件**
  - `phase-01-parse.json`：包含解析后的变更文件和分类结果
  - `phase-02-scan.json`：包含扫描后的图谱数据和诊断信息

#### 检查点管理
- **检查点格式**
  - 包含phaseId、phaseVersion、completedAt、projectRoot等元数据
  - 存储各阶段的核心数据结构
- **数据持久化**
  - 使用compact JSON格式（无缩进）节省空间
  - 自动创建必要的目录结构
- **数据验证**
  - 检查phaseId匹配性
  - 验证时间戳顺序一致性
  - 确认项目根目录一致性

#### 相位验证机制
- **前置条件检查**
  - 验证必需的前置阶段已完成
  - 加载并验证前置阶段的检查点数据
- **数据一致性检查**
  - 检测检查点文件的时间戳顺序
  - 验证项目根目录的一致性
  - 提供警告信息提示潜在问题

#### 大型diff文件处理策略
- **阈值配置**
  - `phasedExecutionThreshold`配置项控制阈值
  - 默认值为1000行，可根据项目规模调整
- **自动切换机制**
  - 当diff行数超过阈值时自动切换到分阶段执行
  - 仅执行parse阶段并生成检查点
  - 后续手动执行scan和analyze阶段

#### 运行目录管理
- **目录结构**
  - `cluster-context/`：存储聚类上下文文件
  - `cluster-analysis/`：存储聚类分析结果
  - 标准化文件命名约定
- **文件组织**
  - 按阶段和聚类ID组织文件
  - 自动生成必要的子目录
  - 支持批量文件操作
